# Coin Hub 交易员策略研究台重构设计

## 1. 背景与目标

当前系统围绕 `BTC`、`ETH` 自动分析控制台构建，包含登录、多页面导航、worker、回放、配置版本、运行快照和旧策略链路。新的目标已经收敛为更小、更直接的研究工具：

- 通过交易员的真实交易记录和行情观点记录沉淀可复用的交易策略
- 在同一个工作台里完成录入、图表查看、样本结算和策略归纳
- 删除当前系统里与这个目标无直接关系的功能
- 本地只用一个命令启动

本次重构将系统定义成一个本地单用户研究台。第一版以手工录入为主，并为后续自动抓取保留明确插槽。

## 2. 产品定义

### 2.1 核心职责

第一版系统只负责四件事：

1. 展示 `BTC` 和 `ETH` 的 K 线
2. 保存交易员资料与交易员记录
3. 将记录转换成可结算样本
4. 基于样本生成候选策略卡片

### 2.2 记录类型

系统支持两类记录：

1. `真实开单记录`
   - 由交易员明确给出开单、关单、方向、价格、时间等信息
   - 系统自动生成执行方案并结算

2. `行情观点记录`
   - 由交易员给出方向性观点或市场判断
   - 用户可以为同一条观点定义多个虚拟开仓方案
   - 每个方案都支持手工补充入场价、出场价、止损、止盈和备注

### 2.3 成功标准

第一版完成后，系统应满足以下结果：

- 一个页面完成录入、查看、结算、归纳闭环
- 真实开单记录可以自动生成样本
- 行情观点记录可以挂多个虚拟执行方案
- 候选策略卡片可以追溯到样本和原始记录
- 本地只需要一个命令启动

## 3. 范围

### 3.1 第一版包含

- `BTC` 和 `ETH` K 线读取与展示
- 交易员资料管理
- 真实开单记录录入
- 行情观点记录录入
- 观点记录的多方案执行计划
- 样本结算
- 候选策略卡片生成与保存
- 记录、样本、策略之间的可追溯关联
- 单页研究台
- 单命令本地启动

### 3.2 第一版明确不包含

- 登录和鉴权
- 多页面控制台
- worker、job queue、scheduler
- replay 实验室
- 配置版本管理
- 运行快照与运行历史
- 自动交易信号生成
- 推文归因链路
- 自动抓取落地实现

## 4. 设计原则

- 研究闭环优先：录入、图表、结算、归纳必须在同一工作台闭环完成
- 记录可追溯：任何策略卡片都必须能反查到样本和原始记录
- 单进程优先：第一版不引入多进程和异步队列
- 最小系统优先：任何与核心链路无直接关系的模块都从主系统退出
- 接口前置：自动抓取先保留来源接口，不进入第一版主流程
- 结构化优先：策略沉淀优先落成结构化卡片，而不是只生成一段总结文本

## 5. 信息架构

### 5.1 单页工作台

系统收缩成一个主工作台页面，推荐结构如下：

1. `记录流与录入区`
   - 交易员筛选
   - 资产切换
   - 记录类型切换
   - 新增记录表单
   - 最近记录时间线

2. `K 线主视图`
   - `BTC / ETH` 切换
   - 周期切换
   - K 线展示
   - 交易员记录时间点标记
   - 执行方案标记
   - 已结算结果标记

3. `当前记录详情区`
   - 交易员信息
   - 原始文本
   - 结构化字段
   - 时间锚点
   - 来源类型

4. `样本结算区`
   - 真实开单自动结算
   - 观点记录多方案定义
   - 手工补充价格和风控
   - 样本状态和失败原因

5. `候选策略卡片区`
   - 策略名称
   - 适用市场环境
   - 触发条件
   - 入场、止损、止盈模板
   - 样本数
   - 胜率
   - 典型案例
   - 手动触发归纳入口

### 5.2 页面规则

- 图表是页面中心视图
- 每条记录都必须锚定到图表时间点
- 候选策略区长期保留在主页面中
- 第一版不拆独立详情页

## 6. 核心数据流

系统主链路固定为：

`trader_record -> execution_plan -> trade_sample -> strategy_candidate`

### 6.1 Trader Record

`trader_record` 表示一条原始记录，必须包含：

- `traderId`
- `symbol`
- `recordType`
- `sourceType`
- `occurredAt`
- `rawContent`
- 可选的结构化交易字段

### 6.2 Execution Plan

`execution_plan` 表示一条可执行解释：

- 真实开单记录默认生成 1 条执行方案
- 行情观点记录可生成多条执行方案
- 每条方案都可独立进入结算
- 方案状态至少包含：`draft`、`ready`、`settled`、`invalid`

### 6.3 Trade Sample

`trade_sample` 表示一条已结算样本，必须包含：

- 方向
- 入场价
- 出场价
- 收益率
- 盈亏结果
- 持仓时长
- 最大回撤
- 结算备注
- 来源记录和方案引用

### 6.4 Strategy Candidate

`strategy_candidate` 表示一条可持续沉淀的候选策略。系统将它定义成独立表，而不是运行时临时聚合结果。这样可以支持后续筛选、修订和人工补充。

候选策略至少包含：

- `name`
- `marketContext`
- `triggerRules`
- `entryRules`
- `riskRules`
- `exitRules`
- `sampleCount`
- `winRate`
- `exampleRefs`

系统同时保留 `strategy_candidate_sample` 关联表，用于维护策略与样本的多对多关系。

第一版的归纳触发方式明确为：

- 由用户在页面中手动触发“归纳候选策略”
- 归纳过程同步执行
- 归纳失败只影响策略卡片区，不影响记录和样本保存

## 8. 模块架构

系统保留单进程架构，模块拆分如下：

### 8.1 market

职责：

- 拉取和标准化 K 线
- 存储和读取本地 candle 数据
- 为图表和结算提供市场数据

可复用当前系统的：

- `src/modules/market-data/*`
- 图表相关能力

### 8.2 records

职责：

- 管理交易员资料
- 管理原始记录
- 管理来源类型和来源元信息

### 8.3 samples

职责：

- 将记录转换成执行方案
- 自动结算真实开单记录
- 结算观点记录的虚拟方案
- 保存样本状态和失败原因

### 8.4 strategies

职责：

- 归纳候选策略
- 保存策略卡片
- 维护策略和样本关系

### 8.5 future-source-adapter

职责：

- 定义自动抓取的来源边界
- 第一版只提供接口定义和手工来源兼容位

## 9. 运行架构

第一版系统是一个本地单进程应用：

- `Next.js App` 提供页面和 API
- `SQLite + Prisma` 提供持久化
- 页面请求直接调用本进程内模块
- 不再存在独立 worker

### 9.1 启动方式

开发态只保留一个命令：

```bash
pnpm dev
```

这个命令应满足：

- 启动 Next 开发服务器
- 自动准备 Prisma Client
- 自动使用本地 SQLite 数据库

系统不再要求：

- `pnpm worker`
- `pnpm scheduler`
- 第二个终端进程

### 9.2 数据刷新

- 页面打开时加载本地 candles
- 页面按固定间隔轮询最新 K 线，第一版使用轮询，不接 WebSocket
- 录入、结算、策略归纳走同步 API
- 后续若出现明显性能瓶颈，再升级为进程内异步

### 10.1 直接删除

- `command-center`

### 10.3 重构顺序

5. 接入记录录入、样本结算和策略归纳

## 11. 错误处理

### 11.1 K 线读取失败

- 页面明确提示不可用资产
- 已有本地 candles 仍可查看
- 缺失时间区间禁止结算新样本

### 11.2 记录不完整

- 真实开单记录可保存为 `待补全`
- 观点方案可保存为 `待定义`
- 未满足结算条件的记录和方案不进入策略统计

### 11.3 样本结算失败

- 样本保留对记录和方案的引用
- 失败原因单独存储
- 用户可修正后重新结算

### 11.4 策略归纳失败

- 不影响记录和样本保存
- 候选策略区显示待重新归纳状态
- 第一版不引入后台重试

- 单命令启动成立

## 14. 后续扩展位

第一版完成后，可在不重写主链路的前提下扩展：

- 自动抓取来源
- 文件导入
- 策略筛选和评分
- 更多资产
- 更复杂的统计分析

这些扩展都建立在同一条主链路之上：

`record -> plan -> sample -> candidate`
